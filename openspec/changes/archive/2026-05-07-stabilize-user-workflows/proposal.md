## Why

近期实际使用暴露出三类高频摩擦：

1. 任务中心失败时经常只显示“分析失败”或泛化建议，用户需要翻日志才能知道是 API Key、模型参数、数据源权限还是真实分析异常。
2. 模拟交易已经支持导入持仓和资金，但多只股票仍需逐行手输，缺少可复用模板和粘贴导入能力。
3. 前端 type-check 存量错误未清零，导致每次前端改动都无法用类型检查证明本次变更是否干净。

这些问题都不属于单一模块功能，而是用户日常工作流的稳定性问题。需要统一成一个小范围改进包，优先把“配置-分析-模拟交易”这条主线变得可诊断、可复用、可验证。

## What Changes

- **任务错误详情增强**
  - 统一任务失败错误结构，保留用户可读摘要、技术细节、建议动作。
  - 前端任务中心“查看错误”优先展示后端真实错误，不再只显示泛化文案。
  - 对常见错误分类：LLM API Key/HTTP 400、数据源权限、行情缺失、未知异常。

- **模拟交易导入模板/粘贴导入**
  - 在“导入持仓和资金”弹窗中增加文本粘贴区。
  - 支持表头或无表头格式：市场、代码、名称、数量、成本价、可用数量。
  - 提供示例模板，并将解析结果落入现有导入表格，提交仍走 `/api/paper/import`。

- **前端 type-check 清零**
  - 修复当前已知 Dashboard 未使用变量错误。
  - 本 change 收口时要求 `npm run type-check -- --pretty false` 退出码为 0。

## Out of Scope

- 不做真实券商导入。
- 不做 CSV 文件上传到外部服务。
- 不重构整个任务系统状态机。
- 不改动真实交易或任何外部系统写入。

## Impact

**改动范围**：`app/routers` / `app/services` 中任务错误结构相关代码、`frontend/src/views/Tasks`、`frontend/src/views/PaperTrading`、`frontend/src/views/Dashboard`、相关 tests。

**风险**：中低。任务错误展示涉及既有失败任务兼容；模拟交易导入解析只在前端本地执行，后端接口复用已存在的导入能力。

**收益**：
- 用户能直接从页面判断失败原因。
- 多持仓迁移更快。
- 前端变更恢复可验证状态。
